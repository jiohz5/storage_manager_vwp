# Storage Manager 관리자 검색 및 사용성 개선 설계

## 목표

RHEL 8.1 MATE 환경에서 대규모 프로젝트 계정 모니터링을 유지하면서 다음 문제를 해결한다.

- 개인 계정 용량을 특정 숫자로 단정하지 않고, 로그인 사용자와 실제 관리 데이터 위치를 명확히 안내한다.
- 느린 `/user` 목록과 대규모 디렉터리 재귀 계산 때문에 최초 경로 선택 창이 멈추는 문제를 제거한다.
- 계정명과 `/user/<계정명>` 경로를 양방향 자동 입력한다.
- inode 의미를 UI에서 바로 이해할 수 있게 한다.
- 추적 탭의 버튼과 열을 줄이고 최근 수집 정보를 넓게 표시한다.
- 창의 `X`를 누르면 GUI만 종료되고 cron과 독립 트레이 알림기는 계속된다는 사실을 알린다.
- 고정 관리자 PIN으로 잠금을 해제한 세션에서만 검색 탭을 표시한다.
- 선택한 프로젝트 계정의 전체 파일명과 디렉터리명을 빠르게 검색할 수 있는 별도 인덱스를 만든다.
- 검색 탭에서 인덱스 DB 전체 크기를 보여 주어 기능 유지 여부를 판단할 수 있게 한다.

## 운영 전제

- Python 3.10 이상과 PyQt5를 사용하며 추가 Python 패키지는 요구하지 않는다.
- 모니터 계정은 읽기와 디렉터리 진입 권한만 필요하다. 프로그램은 `newgrp`, `sg`, `sudo`를 실행하지 않는다.
- 관리 데이터는 모든 계정에 공통인 쓰기 가능한 전용 디렉터리에 둔다.
- cron의 15분 `df` 감시는 상세 스캔과 독립적으로 계속 실행한다.
- 22:00 야간 작업은 시작한 뒤 06:00에 강제 중지하지 않고 완료될 때까지 실행한다.
- 동일 야간 작업은 프로세스 잠금으로 중복 실행하지 않는다.

## 접근법 비교

### 일반 이력 DB에 전체 경로 저장

구조는 단순하지만 수천만 경로가 일반 모니터링 DB와 같은 장애 및 잠금 범위를 공유한다. 검색 기능을 제거하거나 재구축할 때도 이력 DB를 건드려야 하므로 사용하지 않는다.

### 별도 DB를 매일 전체 재작성

삭제된 경로까지 매일 정확히 반영할 수 있지만, 수천만 행의 SQLite 쓰기와 journal 사용량이 반복된다. 파일시스템 순회보다 DB 갱신 부하가 더 커질 수 있어 사용하지 않는다.

### 별도 DB의 재개 가능한 기준선과 일일 변경분 결합

`search_index.db`에 전체 기준선을 만들고, 최초 또는 주기적 전체 대조만 모든 항목을 갱신한다. 그 사이에는 기존 야간 변경 파일 `find` 결과를 같은 순회에서 인덱스에도 반영한다. 일반 모니터링과 장애 범위를 분리하면서 불필요한 이중 순회를 줄이므로 이 방식을 채택한다.

## 데이터 위치 시작 화면

최초 실행은 네이티브 파일 브라우저로 `/user` 전체를 열지 않는다. 작은 전용 대화상자에 경로를 직접 입력하며 다음 내용을 표시한다.

- 현재 로그인 사용자 ID
- 위치 포인터 파일: `$XDG_CONFIG_HOME/storage-manager-vwp/location.json` 또는 `~/.config/storage-manager-vwp/location.json`
- 실제 SQLite, 보고서, 로그, 검색 인덱스가 저장될 관리 데이터 경로
- 개인 계정은 quota가 제한될 수 있으므로 충분한 여유 공간을 선택하라는 안내
- 프로젝트 경로에 전용 디렉터리를 만들 권한이 필요하면 터미널에서 수동 `newgrp` 후 생성하고, 일반 셸로 돌아와 쓰기 검사를 하라는 안내
- 권한이 없으면 다른 쓰기 가능한 여유 공간을 선택하라는 안내

경로 확인은 디렉터리 생성, 작은 파일 쓰기, 임시 SQLite 쓰기만 수행한다. 선택 경로의 전체 재귀 용량 계산은 하지 않는다. 따라서 수십 TB 프로젝트 루트를 잘못 입력해도 `Open` 직후 전체 트리를 읽지 않는다.

## 계정 입력

- 이름을 사용자가 편집하면 첫 번째 허용 루트를 prefix로 사용해 `/user/<이름>` 형태를 경로 칸에 채운다.
- 경로를 사용자가 편집하면 마지막 경로 요소를 계정명 칸에 채운다.
- 프로그램의 `setText`로 발생한 변경은 다시 반대 필드를 갱신하지 않아 순환 신호를 막는다.
- 실제 등록 시 기존과 동일하게 허용 루트 바로 아래의 존재하고 읽을 수 있는 디렉터리인지 검증한다.
- 계정 탭은 모니터링 자체에는 쓰기 권한이나 `newgrp`가 필요하지 않고 관리 데이터는 별도 전역 경로에 저장된다고 설명한다.

## inode 안내

한국어 열 이름은 `파일 수 한도 (inode)`로 표시한다. 헤더 툴팁은 inode가 파일과 폴더마다 하나씩 소비되며, byte 용량이 남아도 inode가 100%이면 새 파일을 만들 수 없다고 설명한다. 영어 UI도 같은 의미의 툴팁을 제공한다.

## 추적 화면

추적 탭의 작업 버튼은 다음 네 개 역할로 줄인다.

- 새로고침
- 자동 추적 켜기/끄기: cron 설치 상태에 따라 한 버튼의 의미가 바뀐다.
- 야간 작업 실행/안전 중지: 프로세스 상태에 따라 한 버튼의 의미가 바뀐다.
- 팝업 알림 시작/중지: 독립 트레이 알림기 상태에 따라 한 버튼의 의미가 바뀐다.

로그인 시 알림기 자동 시작은 체크박스로 표현한다. 재시작, 설치, 제거처럼 같은 대상을 조작하는 별도 버튼들은 제거한다.

계정별 표는 `계정`, `경로`, `추적 상태`, `용량/파일 수/quota`, `100% 예상`, `최근 수집`, `기준선`으로 축약한다. `경로`와 `최근 수집` 열이 남는 폭을 공유하며, 기본 창 크기는 VWP 화면에서 전체 버튼이 보이도록 줄인다.

## 종료 동작

`X`를 누르면 메인 GUI를 실제로 종료한다. 종료 전에 다음 상태를 한 번 안내한다.

- cron 등록 여부와 관계없이 GUI 프로세스가 종료된다는 점
- 등록된 cron 작업은 계속된다는 점
- 독립 트레이 알림기가 실행 중이면 MATE 패널의 디스크 아이콘에서 상태를 확인하고 관리 창을 다시 열 수 있다는 점
- 트레이 알림기가 실행 중이 아니면 팝업은 오지 않지만 cron 수집 자체는 계속될 수 있다는 점

메인 GUI를 트레이에 숨겨 계속 실행하지 않는다.

## 관리자 모드

상단 `관리자` 메뉴는 잠금 상태에서 `관리자 모드 열기`를 제공한다. 비밀번호 입력은 마스킹하며 고정 PIN의 평문 대신 소스에 검증용 PBKDF2-HMAC-SHA256 결과를 저장하고 `hmac.compare_digest`로 비교한다.

올바른 PIN을 입력하면 현재 GUI 세션에만 관리자 권한을 부여하고 `검색` 탭을 동적으로 추가한다. `관리자 모드 잠금`을 선택하거나 GUI를 종료하면 검색 탭이 다시 숨겨진다. 이 방식은 소스와 데이터 디렉터리에 접근할 수 있는 사용자를 막는 강한 보안 경계가 아니라 소수 사용자 사이에서 기능을 숨기는 UI 잠금이다.

## 검색 인덱스 저장소

검색 인덱스는 관리 데이터 디렉터리의 `search_index.db`에 저장하고 일반 `storage_manager.db`와 분리한다. SQLite는 네트워크 파일시스템 호환성을 위해 기본 journal 모드를 사용하고 짧은 트랜잭션과 `busy_timeout`을 적용한다.

주요 테이블은 다음과 같다.

- `search_entries`: 계정 ID, 계정 기준 상대 경로, basename, 확장자, 항목 종류, 현재 전체 스캔 generation
- `search_scan_state`: 계정 경로, 상태, generation, 시작/갱신/완료 시각, 파일 수, 디렉터리 수, 오류
- `search_scan_tasks`: 계정과 generation별 아직 방문하지 않은 상대 디렉터리 작업

검색 행에는 절대 경로를 반복 저장하지 않는다. 계정 경로와 상대 경로를 GUI에서 조합해 DB 크기를 줄인다. 심볼릭 링크 자체는 검색 결과로 기록할 수 있지만 링크 대상 디렉터리로는 내려가지 않으며 다른 파일시스템으로 넘어가지 않는다.

`account_id + relative_path`를 기본 키로 사용한다. 계정별 basename, 확장자, 항목 종류 검색 인덱스를 별도로 생성한다.

## 전체 인덱싱

관리자 검색 탭에서 계정별로 검색 인덱싱을 활성화한다. 기본값은 꺼짐이다.

전체 기준선은 Python `os.scandir`로 디렉터리 하나씩 처리한다. 디렉터리 작업을 완료할 때마다 항목 upsert, 하위 디렉터리 작업 추가, 체크포인트 갱신을 하나의 짧은 트랜잭션으로 저장한다. 프로세스가 중지되면 다음 야간 실행이 남은 디렉터리부터 이어간다.

전체 generation이 끝나면 이전 generation에만 남은 항목을 삭제해 실제로 사라진 파일과 디렉터리를 정리한다. 최초 전체 인덱스와 마지막 완전 인덱스 이후 7일이 지난 계정만 전체 대조 대상이다.

## 일일 변경 반영

기존 `scan_changed_file_activity`가 `find` 출력에서 파일 경로를 읽을 때, 검색 인덱싱이 활성화된 계정이면 같은 스트림의 경로를 일정 크기 배치로 `search_index.db`에 upsert한다. 별도 `find`를 한 번 더 실행하지 않는다.

생성되거나 수정된 파일과 그 상위 디렉터리는 일일 변경 반영으로 검색 가능해진다. 삭제와 일부 rename은 `find -newermt`만으로 완전히 판별할 수 없으므로 최대 7일 뒤 전체 대조에서 정리된다. 검색 탭은 마지막 완전 인덱스 시각을 표시해 이 snapshot 성격을 숨기지 않는다.

## 야간 실행 정책

- cron은 22:00에 하나의 야간 프로세스를 시작한다.
- 빠른 용량 수집과 보고서 생성을 먼저 완료한 뒤 필요한 검색 기준선 작업을 수행한다.
- 06:00에 새 작업을 막거나 실행 중 작업을 강제 종료하지 않는다.
- 수동 `안전 중지` 요청은 현재 디렉터리 배치를 저장한 뒤 종료하며 다음 밤에 재개한다.
- 다음 22:00까지 이전 프로세스가 남아 있으면 공통 lock 때문에 중복 작업은 시작하지 않는다.
- `nice`와 `ionice`는 사용 가능할 때 시작 명령에 적용하되, 지원되지 않거나 네트워크 스토리지에서 효과가 없어도 기능은 계속 동작한다.

## 검색 UI

검색 탭은 관리자 잠금 해제 후에만 존재한다. 한 번에 한 계정을 선택하며 다음을 제공한다.

- 계정별 인덱싱 활성화/비활성화
- 선택 계정의 상태, 파일 수, 디렉터리 수, 마지막 완전 인덱스, 마지막 변경 반영 시각
- 전체 `search_index.db`와 관련 journal 파일의 합산 크기
- 전체 인덱스 항목 수와 선택 계정 인덱스 항목 수
- 이름 검색 모드: 정확히 일치, prefix, 포함
- 확장자 정확 검색
- 항목 종류: 전체, 파일, 디렉터리, 링크
- 최대 500개 결과와 전체 경로 표시

정확 일치, prefix, 확장자 검색은 SQLite 인덱스를 사용한다. 이름 포함 검색은 `%문자열%` 방식이라 대규모 DB에서 상대적으로 느릴 수 있음을 UI에 표시한다. 검색 자체는 GUI worker thread에서 수행해 잠금 대기나 느린 포함 검색 중에도 창이 멈추지 않게 한다.

## 용량 표시

검색 탭의 DB 크기는 `search_index.db`, `search_index.db-journal`, `search_index.db-wal`, `search_index.db-shm` 중 존재하는 파일의 합으로 계산한다. 새로고침할 때마다 값과 항목 수를 갱신한다.

실험용 스키마에서 평균 경로 길이를 포함한 측정치는 약 497 byte/항목이었다. UI 안내에는 실제 값이 경로 길이와 파일시스템 구성에 따라 달라진다는 점과 함께 대략 다음 기준을 표시한다.

- 100만 항목: 약 0.5 GB
- 1,000만 항목: 약 4.6 GB
- 5,000만 항목: 약 23 GB

이 값은 경고용 추정치이고 유지 여부 판단은 검색 탭에 표시되는 실제 DB 크기를 우선한다.

## 오류 처리

- 읽을 수 없는 디렉터리는 해당 작업을 오류로 기록하고 전체 프로세스를 무한 재시도하지 않는다.
- 잘못된 UTF-8 파일명은 Python filesystem decoding 결과에서 SQLite에 저장 가능한 대체 문자로 정규화하고 카운트에 경고를 남긴다.
- 검색 DB 잠금은 `busy_timeout` 후 사용자에게 `인덱싱 중이므로 잠시 후 다시 검색` 메시지로 표시한다.
- 검색 인덱스 오류는 일반 용량 수집과 보고서 생성을 실패시키지 않는다.
- 계정 경로가 변경되면 해당 계정의 이전 검색 인덱스와 작업을 폐기하고 새 기준선을 시작한다.

## 테스트 전략

- 런타임 테스트: 경로 검사에서 재귀 크기 계산을 생략하는지, 사용자 ID와 포인터 경로를 생성하는지 확인한다.
- GUI 테스트: 직접 경로 입력, 이름/경로 양방향 자동 입력, inode 툴팁, 단순화된 추적 버튼, 종료 안내를 검증한다.
- 인증 테스트: 올바른 고정 PIN만 통과하고 잘못된 값은 거절되며 평문 PIN 상수가 없는지 확인한다.
- 검색 DB 테스트: 정확, prefix, 포함, 확장자, 종류 검색과 DB 크기 합산을 검증한다.
- 인덱서 테스트: 중단 후 디렉터리 체크포인트 재개, generation 완료 후 삭제 항목 정리, 심볼릭 링크 비추적, 계정 경로 변경 재구축을 검증한다.
- 통합 테스트: 기존 변경 파일 스트림이 검색 DB를 배치 갱신하고, 검색 오류가 일반 야간 보고서를 막지 않는지 확인한다.
- 전체 회귀 테스트와 Python compile 검사를 실행한다.

## 배포 및 호환성

새 모듈과 테스트는 소스 번들에 포함한다. 검색 인덱스와 상태 파일은 `.gitignore` 및 번들 제외 목록에 포함해 실제 프로젝트 파일명이 GitHub나 배포 압축에 들어가지 않게 한다.

Windows에서는 임시 디렉터리와 가짜 backend로 로직을 테스트한다. 실제 csh, cron, MATE 트레이, `nice`/`ionice`, 수천만 파일 성능은 RHEL VWP 환경에서 최종 확인한다.
