# Code Review

## 수정된 주요 문제

### P0: GUI 새로고침마다 SQLite 행이 무제한 증가

기존 코드는 60초마다 계정별 새 행을 추가했습니다. 20개 계정이면 1년에 약
1,051만 행이 생성되어, 저장공간을 지키려는 프로그램이 자체 DB로 공간을
사용하는 구조였습니다.

현재는 `day + account_id + source` 단위로 upsert합니다. GUI와 야간 작업을
합쳐도 계정별 하루 최대 2행이며 365일 후 정리됩니다.

### P0: `du` 제한시간이 계정이 아닌 하위 항목마다 적용

기존에는 top-level 항목마다 최대 1시간을 허용했습니다. 항목이 100개라면 계정
하나가 최악의 경우 100시간을 점유할 수 있었습니다.

현재는 모든 계정의 `df`를 먼저 수집한 뒤 계정별 최대 1시간, 개별 디렉터리별
최대 15분으로 제한합니다. 15분을 넘긴 디렉터리는 한 단계씩 나누고 완료 작업을
SQLite에 체크포인트로 남겨 다음 밤에 이어갑니다. 상세 작업은 22:00~06:00 창 안에서
실행하며 05:45 이후로 넘어가지 않도록 안전 여유시간을 둡니다.

### P1: 네트워크 파일시스템 지연 시 GUI 정지

기존에는 GUI 메인 스레드에서 `df`를 순차 실행했고 timeout도 없었습니다. 현재는
최대 4개 백그라운드 작업으로 실행하며 각각 기본 15초 제한시간을 사용합니다.

### P1: 증가량 계산 오류

기존에는 전날 상위 15개와 오늘 상위 15개만 비교했습니다. 전날 16위였던 기존
폴더가 오늘 15위가 되면 전체 폴더 크기가 신규 증가량으로 표시됐습니다.

현재는 전체 top-level 최신 기준선을 별도 테이블에 한 세트만 저장하고 그 기준선과
비교합니다. 첫 스캔은 증가량이 아니라 기준선 설정으로 표시됩니다.

### P1: 임의 경로 등록 가능

기존에는 존재 여부와 범위를 확인하지 않아 `/`, 개인 홈, 심볼릭 링크를 통한 외부
경로도 등록할 수 있었습니다. 현재는 실제 경로를 해석한 후 읽을 수 있는
`/user` 바로 아래 디렉터리만 허용합니다.

### P1: 야간 작업 중복 실행

상세 스캔이 다음 날까지 지속될 때 새 cron 작업이 겹칠 수 있었습니다. 현재는
프로세스 파일 잠금으로 하나의 야간 작업만 실행합니다.

### P2: 설정 파일과 로그 내구성

설정 JSON은 임시 파일, flush, fsync, replace 순서로 저장합니다. cron 로그는
5MB 단위로 최대 3개까지 회전하며, 보고서와 DB 이력은 보존기간 후 정리됩니다.

### P1: PID 강제 종료 위험

추적 화면에서 lock 파일의 숫자만 보고 signal을 보내면 PID 재사용으로 무관한 작업을
종료할 수 있습니다. 현재는 실행별 run ID가 일치하는 중지 요청 파일만 사용하고,
RHEL에서는 `/proc/<pid>/cmdline`의 스크립트와 데이터 경로도 확인합니다. `du/find`가
요청을 감지하면 해당 subprocess를 정리하고 완료된 기준선 체크포인트를 보존합니다.

## 현재 운영상 의미

- `df` 값은 계정 자체 사용량이 아니라 해당 경로가 속한 파일시스템 사용량입니다.
  이는 요청한 `cd /user/account; df .`와 동일한 의미입니다.
- 계정별 상세 사용량과 증가 경로는 완료된 rolling `du` 기준선끼리 비교한 값입니다.
- 기준선 완료 직전 top-level을 다시 대조하고, 기준선 시작 이후 변경 파일은 다음
  `find -newermt` 검사에서 다시 확인합니다.
- 일일 변경 byte는 수정된 파일의 현재 크기 합계이지 순증감이 아닙니다. 삭제와
  순증감은 완료된 주간 기준선에서 보정합니다.
- timeout이나 권한 오류가 발생하면 해당 날짜의 상세 기준선은 갱신하지 않습니다.
- 모니터링 대상 파일은 자동 삭제하지 않습니다. 보고서와 경고만 제공합니다.
- 폐쇄망 알림은 기본 outbox와 선택적 shell 없는 사내 command·내부 webhook으로
  분리했습니다. 실제 endpoint를 설정하지 않으면 외부 전송은 발생하지 않습니다.

## 다음 향상 후보

1. 여러 관리자가 하나의 이력을 공유해야 한다면 네트워크 SQLite 공유 대신 중앙
   수집 프로세스와 읽기 전용 GUI 구조가 적합합니다.
2. `find` 메타데이터 순회도 제한시간을 반복해서 넘는 파일시스템이라면, 실제 스토리지
   종류에 맞는 quota/snapshot/change-log API가 있는지 확인하는 것이 우선입니다.

- `/user/<account>`에서 `df -Pk`, `df -Pi`, `du`, `find -newermt -printf`가 허용되는지
- `crontab` 사용 권한과 22시 수집·07시 건강 점검 정책
- quota JSON wrapper와 사내 command/webhook endpoint 형식
- 데이터 디렉터리가 한 관리자 전용인지
